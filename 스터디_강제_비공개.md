## 1. 기능 개요

### 1.1. 무엇을 하는 기능인가

플랫폼 관리자가 스터디 관리 페이지(`/studies`) 에서 부적절한 스터디를 **강제로 비공개 처리**할 수 있다. 강제 비공개된 스터디는:

- `closed = true` 로 설정되어 일반 사용자 화면에서 차단된다.
- `published` 값은 **그대로 유지**된다 (공개됐던 이력 보존).
- 한 번 처리하면 **되돌릴 수 없다** (일방향 전환).

### 1.2. 왜 이 기능이 학습 가치가 높은가

회원 권한 변경과 동일한 5단계 호출 패턴을 따르면서도, **도메인 설계 측면에서 근본적으로 다른 문제**를 다룬다.

| 주제 | 회원 권한 변경 | 스터디 강제 비공개 |
|------|----------------|------------------|
| 상태 전이 방향 | **양방향 토글** (USER ↔ ADMIN) | **일방향 전환** (→ closed) |
| 자기 자신 검증 | 필요 (관리자 전멸 위험) | **불필요** (자기 스터디도 합법적 차단 가능) |
| 도메인 불변식 | 없음 (role 은 단순 필드) | **있음** (State Machine — 이미 종료된 스터디 재종료 금지) |
| 상태 조합 | 2가지 (USER / ADMIN) | **4가지** (비공개 / 공개 / 공개+모집중 / 종료됨) |
| 요청 본문 | `{ role: "..." }` | **없음** (단일 동작) |

이 기능을 통해 **"도메인 규칙이 엔티티 메서드 안에 어떻게 살아있는가"** 를 실감할 수 있다.

## 3. Option A — 상태 전이 설계 결정

강제 비공개를 구현할 때 **세 가지 상태 전이 옵션**이 후보에 있었다. 어느 옵션을 선택하는가가 이 기능의 성격을 결정한다.

### 3.1. 세 가지 후보

```
[Option A] 최종 상태 = 일반 close() 와 동일
           closed=true, published=유지
           +  엔티티에 forceClose() 메서드 추가 (published 전제조건 없음)
           
[Option B] 최종 상태 = 완전 미공개
           closed=true, published=false 로 되돌림
           → "공개된 적이 있다는 이력" 소실
           
[Option C] 새 상태 플래그 추가
           closed=true, published=유지, forcedClosed=true (신규 필드)
           + 엔티티에 forcedClosed 필드 추가
           + DB 스키마 변경
```

### 3.2. 왜 Option A 를 선택했는가

| 기준 | Option A | Option B | Option C |
|------|----------|----------|----------|
| 엔티티 변경 | 메서드 1개 추가 | 메서드 1개 추가 | **필드 + 메서드 + 마이그레이션** |
| State Machine 불변식 | 유지 ✅ | **깨짐** (published 역전) | 유지 |
| 이력 보존 | 유지 ✅ | **소실** | 유지 |
| "관리자가 닫았음" 구분 | ❌ (나중에 필요하면 C로) | ❌ | ✅ |
| 구현 복잡도 | 낮음 | 중간 | **높음** |

**결정 이유**: 지금 당장 "관리자가 닫았음" 배지가 필요한 것은 아니다. 나중에 그 요구가 생기면 Option C 로 확장하면 된다 — 그때 `forcedClosed boolean` 필드를 추가하고 `forceClose()` 내부에 `this.forcedClosed = true` 한 줄만 추가하면 된다.

**YAGNI (You Aren't Gonna Need It) 원칙**: 지금 필요 없는 기능은 만들지 않는다. 확장 포인트만 열어둔다.

### 3.3. forceClose() 와 일반 close() 의 두 가지 차이

```java
// Study.java 에서 발췌

public void close() {
    if (this.published && !this.closed) {
        this.closed = true;
        this.closedDateTime = LocalDateTime.now();
    } else {
        throw new RuntimeException("스터디를 종료할 수 없습니다. " +
            "스터디를 공개하지 않았거나 이미 종료한 스터디입니다.");
    }
}

public void forceClose() {
    if (this.closed) {
        throw new RuntimeException("이미 종료된 스터디는 다시 종료할 수 없습니다.");
    }
    this.closed = true;
    this.closedDateTime = LocalDateTime.now();
}
```

**차이 1 — `published` 전제조건**

| 메서드 | published=false 인 스터디 처리 |
|--------|-------------------------------|
| `close()` | ❌ 예외 ("공개하지 않은 스터디는 종료할 수 없음") |
| `forceClose()` | ✅ 허용 |

**왜 차이가 있는가**: 소유자가 자기 스터디를 "종료" 한다는 것은 "공개된 것을 마감한다" 는 뜻이다. 공개도 안 한 스터디를 "종료" 하는 것은 의미가 없다.

그러나 관리자는 **공개 전 스터디라도 부적절하다고 판단하면 차단**할 수 있어야 한다. 예: 회원가입 직후 욕설만 가득한 스터디를 만들고 공개 버튼만 안 누른 경우. 그 스터디가 영원히 "draft" 상태로 남아 있으면 플랫폼 관리 측면에서 구멍이 된다.

**차이 2 — 이미 종료된 경우 동작**

| 메서드 | closed=true 인 스터디 처리 |
|--------|---------------------------|
| `close()` | ❌ 예외 |
| `forceClose()` | ❌ 예외 (다른 메시지) |

둘 다 예외를 던지지만 이유가 다르다:
- `close()`: 이미 종료된 스터디에 대한 중복 종료 방지 (호출자가 실수한 경우)
- `forceClose()`: 관리자에게 "이미 다른 관리자가 처리했다" 를 명확하게 알림

**왜 "조용히 무시" 하지 않는가**: 성공 응답을 주면서 아무 변화가 없으면, 관리자는 **"뭔가 된 것 같은데"** 라고 착각하게 된다. 예외로 명확하게 실패를 알리는 것이 **정직한 설계**다.

## 4. 회원 권한 변경과의 차이점

두 기능은 **5단계 네트워크 호출 패턴을 공유**하지만, 비즈니스 규칙과 UX 에서 다르다.

### 4.1. 자기 자신 검증 없음

```java
// admin-service AdminMemberController (회원 권한 변경)
if (requesterId.equals(id)) {
    return ResponseEntity.badRequest()
            .body(CommonApiResponse.fail("자기 자신의 권한은 변경할 수 없습니다."));
}

// admin-service AdminStudyController (스터디 강제 비공개)
// ← 자기 자신 검증이 없다
```

**왜 없는가**: "관리자가 자기 스터디를 강제 종료하는 것" 은 **합법적인 행위**다. 플랫폼 관리자는 자기가 만든 스터디라도 문제가 있다고 판단하면 닫을 수 있어야 한다 (자기 반성).

회원 권한 변경은 "관리자가 자기 권한을 강등" 하면 **시스템의 모든 관리자가 사라지는 위험**이 있어 절대 차단. 스터디는 그런 전역 위험이 없다.

### 4.2. 요청 본문 없음

```java
// 회원 권한 변경 — 본문 있음
fetch('/members/123/role', {
    method: 'POST',
    body: JSON.stringify({ role: 'ROLE_ADMIN' })   // body 필요
});

// 스터디 강제 비공개 — 본문 없음
fetch('/studies/spring-boot-study/force-close', {
    method: 'POST'
    // body 없음
});
```

**왜 없는가**: "강제 종료" 라는 **단일 동작**만 수행하므로 추가 파라미터가 필요 없다. URL 경로 (`/force-close`) 에 의도가 모두 담겨 있다.

이것이 RESTful 관점에서 정당하다 — 상태 전이를 의도한 POST 는 **본문이 필수가 아니다** (HTTP 스펙 RFC 7231).

### 4.3. 일방향 전환 (토글 ≠ 일방향)

```javascript
// 회원 권한 변경 — 토글
// ROLE_USER ↔ ROLE_ADMIN 자유롭게 오감
function applyRoleChange(row, newRole) {
    // 새 role 에 맞게 버튼 라벨 바꿔가며 계속 토글
    var nextRole = isAdmin ? 'ROLE_USER' : 'ROLE_ADMIN';
    var nextLabel = isAdmin ? '사용자로 강등' : '관리자로 승격';
    // 버튼을 새로 그리고 다음 클릭을 대기
}

// 스터디 강제 비공개 — 일방향
function applyForceClose(row) {
    // 버튼 자체를 제거하고 "이미 종료됨" 텍스트로 대체
    actionsCell.innerHTML =
        '<span class="muted-text">이미 종료됨</span>';
    row.classList.add('row-closed');  // 행 흐리게 처리
}
```

**이 차이가 UX 에 미치는 영향**: 강제 비공개는 **되돌릴 수 없는 위험한 작업**이므로 회원 권한 변경보다 훨씬 강한 경고를 보여준다.

```javascript
// 회원 권한 변경 확인 다이얼로그
confirm(nickname + ' 님을 ' + roleLabel + ' 로 변경하시겠습니까?');

// 스터디 강제 비공개 확인 다이얼로그
confirm('「' + title + '」 스터디를 강제 비공개 처리하시겠습니까?\n\n' +
        '이 작업은 되돌릴 수 없으며, 스터디가 종료 상태가 됩니다.');
```

### 4.4. 복잡한 상태 배지 조합

```html
<!-- 회원 권한 변경 — 상태 배지 1개 -->
<td class="cell-role">
    <span class="badge badge-admin">ADMIN</span>  <!-- 또는 USER -->
</td>

<!-- 스터디 강제 비공개 — 상태 배지 조합 -->
<td class="cell-status">
    <span th:if="${s.closed}" class="badge badge-closed">종료됨</span>
    <span th:unless="${s.closed}" th:if="${s.published}" class="badge badge-published">공개</span>
    <span th:unless="${s.closed or s.published}" class="badge badge-draft">비공개</span>
    <span th:if="${s.recruiting and not s.closed}" class="badge badge-recruiting">모집중</span>
</td>
```

**왜 배지를 조합하는가**: 스터디는 **여러 상태가 조합**된 State Machine 을 가진다.

| 조합 | 의미 | 표시 |
|------|------|------|
| `!published && !closed` | 작성 중 (비공개) | "비공개" |
| `published && !recruiting && !closed` | 공개됐지만 모집 종료 | "공개" |
| `published && recruiting && !closed` | 공개 + 모집중 | "공개" + "모집중" |
| `closed` | 종료됨 (published 무관) | "종료됨" |

이 조합 로직이 HTML 에서 직접 분기되어 있다. Thymeleaf `th:if` / `th:unless` 의 조합으로 State Machine 을 그대로 UI 에 반영한다.

## 5. 계층별 상세 구현

### 5.1. study-service — 도메인 소유 계층

#### 5.1.1. `Study.forceClose()` — 도메인 메서드

```java
// Study.java 에서 발췌
public void forceClose() {
    if (this.closed) {
        throw new RuntimeException("이미 종료된 스터디는 다시 종료할 수 없습니다.");
    }
    this.closed = true;
    this.closedDateTime = LocalDateTime.now();
}
```

**왜 `setClosed(true)` 가 아닌가**

회원 권한 변경의 `changeRole()` 과 같은 이유이지만, **더 강한 이유**가 있다. Study 의 상태는 State Machine 이어서 여러 필드가 함께 변해야 한다:

- `closed = true` ← 종료 표시
- `closedDateTime = now()` ← 종료 시각 기록

`setClosed(true)` 만 호출하면 `closedDateTime` 이 null 로 남는다. 도메인 메서드로 감싸면 **"종료한다"** 라는 비즈니스 행위에 필요한 모든 필드 변경을 원자적으로 수행한다.

**불변식(invariant) 검사**

```java
if (this.closed) {
    throw new RuntimeException("이미 종료된 스터디는 다시 종료할 수 없습니다.");
}
```

이 한 줄이 **State Machine 의 핵심**이다. 이미 `closed=true` 인 상태에서 `forceClose()` 가 또 호출되면 `closedDateTime` 이 **두 번째 호출 시각으로 덮어쓰여진다**. 첫 종료 시각 정보가 손실된다.

이 불변식을 서비스 계층에서 검사할 수도 있지만, **도메인 객체 본인이 검사하는 편이 훨씬 견고**하다. 왜냐하면:

1. 미래에 어떤 서비스가 이 메서드를 호출할지 모른다.
2. 모든 호출자가 검사를 잊지 않도록 강제할 방법이 없다.
3. 객체가 **자기 상태를 스스로 지키는 것**이 객체지향의 핵심이다.

**왜 `publishedDateTime` 은 건드리지 않는가**

```java
public void forceClose() {
    if (this.closed) { throw ... }
    this.closed = true;
    this.closedDateTime = LocalDateTime.now();
    // publishedDateTime 은 절대 건드리지 않음
    // published 필드도 그대로 유지
}
```

공개된 적이 있는 스터디라면 그 시점의 기록은 보존하는 것이 맞다. "공개됐었지만 관리자에 의해 종료됨" 이라는 이력이 둘 다 의미 있는 정보이기 때문이다.

이것이 Option A 의 핵심 — **이력 보존**.

---

#### 5.1.2. `StudyRepository` — 관리자용 검색 메서드

```java
Page<Study> findByTitleContainingIgnoreCaseOrPathContainingIgnoreCase(
        String titleKeyword, String pathKeyword, Pageable pageable);
```

**왜 기존 공개 목록 메서드를 재사용하지 않는가**

기존 `findFirst9ByPublishedAndClosedOrderBy...` 같은 메서드들은 **"published=true, closed=false"** 조건을 고정해서 건다. 일반 사용자가 보는 "공개된 활동 중 스터디 목록" 을 위한 것이다.

관리자는 **미공개·종료된 스터디까지 전부** 봐야 한다. 따라서 조건 없이 전체를 조회해야 한다. 별도 메서드가 필요한 이유.

**Spring Data JPA 가 메서드 이름으로 쿼리를 만드는 방식**

```
findBy                         → SELECT
TitleContainingIgnoreCase      → WHERE LOWER(title) LIKE LOWER(CONCAT('%', ?, '%'))
Or                             → OR 로 조건 결합
PathContainingIgnoreCase       → WHERE LOWER(path) LIKE LOWER(CONCAT('%', ?, '%'))
```

실행되는 쿼리:

```sql
SELECT * FROM study
WHERE LOWER(title) LIKE LOWER('%키워드%')
   OR LOWER(path)  LIKE LOWER('%키워드%')
ORDER BY ...                    -- Pageable 의 Sort
LIMIT ? OFFSET ?                -- Pageable 의 page/size
```

**왜 `IgnoreCase` 를 붙였는가**

- `path` 는 소문자·하이픈으로만 구성되어 원래 대소문자 이슈 없음.
- `title` 은 한글·영문·숫자가 섞여 있어 대소문자 구분이 검색에 방해됨.

관리자가 "Spring" 으로 검색했는데 `"spring-boot-study"` 만 보이고 `"Spring Boot 스터디"` 는 놓치면 혼란스럽다. `IgnoreCase` 로 대소문자 구분을 없애면 **한 번의 검색으로 둘 다 잡힌다**.

**null-safe 하지 않은 점 주의**

`keyword` 가 `null` 로 들어오면 `LIKE '%null%'` 같은 이상한 쿼리가 만들어지거나 오류가 난다. 호출 측 (Service) 에서 `null`/빈 문자열을 `findAll(pageable)` 로 분기시키는 것이 안전하다.

```java
// StudyInternalController 에서 이 분기 처리
if (keyword == null || keyword.isBlank()) {
    page = studyRepository.findAll(pageable);
} else {
    page = studyRepository.findByTitleContainingIgnoreCaseOrPathContainingIgnoreCase(
            keyword, keyword, pageable);
}
```

#### 5.1.3. `StudyAdminResponse` — 관리자 전용 DTO

```java

@Builder
@NoArgsConstructor
@AllArgsConstructor
public class StudyAdminResponse {

    private Long id;
    private String path;
    private String title;
    private String shortDescription;

    private boolean published;
    private boolean closed;
    private boolean recruiting;

    private int memberCount;

                .memberCount(study.getMemberCount())

                .closedDateTime(study.getClosedDateTime())
                .build();

```

**왜 기존 DTO 를 재사용하지 않고 별도로 만들었는가**

| 기존 DTO | 용도 | 문제점 |
|----------|------|--------|
| `StudySummaryResponse` | 공개 카드 목록용 | `closed`/`published` 같은 관리 상태 정보 부족, 외부 노출 전제 설계 |
| `StudyInternalResponse` | 상세 페이지용 | `managerIds`/`memberIds` 전체 컬렉션 포함 → 목록 조회에 너무 무거움 + `@ElementCollection` 추가 쿼리 |

관리자 목록은 **"상태를 한눈에 보고 판단한다"** 라는 고유 목적이 있다. 그 용도에 최적화된 필드만 담는 **전용 DTO 가 가장 깔끔**하다. 앞으로 "신고 건수" 같은 관리자 전용 필드가 추가되어도 공개 DTO 를 건드리지 않고 이 DTO 만 확장하면 된다.

**왜 `memberCount` 는 포함하지만 `memberIds` 는 포함하지 않는가**

목록 화면에서 **"멤버가 몇 명인지"** 는 중요하지만, **"누가 멤버인지"** 는 행마다 수십 개의 id 를 내려주는 부담만 늘린다.

`memberCount` 는 Study 엔티티에 **비정규화 필드**로 저장되어 있어 컬렉션 테이블 (`study_member_ids`) 에 접근할 필요도 없다. `study.addMember()` / `removeMember()` 가 호출될 때마다 이 값이 함께 갱신된다.

**비정규화의 트레이드오프**:
- ✅ 장점: 목록 조회 쿼리가 간단해짐 (JOIN 불필요)
- ❌ 단점: 코드 변경 시 불일치 위험 (메서드를 건너뛰고 memberIds 만 직접 수정하면 count 가 안 맞음)

이 트레이드오프를 받아들인 이유: **목록 조회 성능이 압도적으로 중요**. 일관성은 도메인 메서드를 반드시 경유하는 규율로 보완.

#### 5.1.4. `StudyInternalService.forceClose()` — 비즈니스 로직

```java
@Service
@RequiredArgsConstructor
public class StudyInternalService {

    private final StudyRepository studyRepository;

    @Transactional
    public StudyAdminResponse forceClose(String path, Long requesterId) {

        // 가드 1 — 요청자 헤더 필수
        if (requesterId == null) {
            throw new IllegalArgumentException("요청자 정보가 없습니다.");
        }

        // 가드 2 — 대상 스터디 존재 검증
        Study study = studyRepository.findByPath(path);
        if (study == null) {
            throw new IllegalArgumentException("스터디를 찾을 수 없습니다: path=" + path);
        }

        // 가드 3 — 도메인 메서드 호출 + 예외 래핑
        try {
            study.forceClose();
        } catch (RuntimeException e) {
            throw new IllegalArgumentException(e.getMessage());
        }

        return StudyAdminResponse.from(study);
    }
}
```

**왜 자기 자신 검증이 없는가** (재확인)

회원 권한 변경의 `AccountInternalService.updateRole()` 에는 다음 가드가 있었다:

```java
if (requesterId.equals(targetId)) {
    throw new IllegalArgumentException("자기 자신의 권한은 변경할 수 없습니다.");
}
```

스터디 강제 비공개에는 이런 가드가 **의도적으로 없다**. 관리자가 자기 스터디를 닫는 것은 합법적이다.

**예외 래핑의 의미 — `RuntimeException` → `IllegalArgumentException`**

```java
try {
    study.forceClose();
} catch (RuntimeException e) {
    throw new IllegalArgumentException(e.getMessage());
}
```

Study 엔티티의 `forceClose()` 는 `RuntimeException` 을 던진다 ("이미 종료된 스터디는..."). 이를 그대로 밖으로 내보내면 `GlobalExceptionHandler` 가 500 Internal Server Error 로 처리할 수 있다.

**`IllegalArgumentException` 으로 래핑**하면 의미가 달라진다:

- `RuntimeException` (엔티티 내부 어디선가 발생) → 500 Internal Server Error
- `IllegalArgumentException` (Service 가 명시적으로 던짐) → 400 Bad Request

두 예외 모두 `RuntimeException` 의 자식이므로 Spring 트랜잭션 롤백 정책은 동일하다. 다른 점은 **의미론**이다. "사용자가 잘못 요청한 것" 임을 400 으로 명확히 드러낸다.

**엔티티 메시지 보존**

```java
throw new IllegalArgumentException(e.getMessage());  // "이미 종료된 스터디는..."
```

엔티티에서 작성한 **사용자 친화적 메시지**를 그대로 전달한다. Service 가 새로 메시지를 만들면 엔티티의 맥락이 소실된다. 래핑만 하고 메시지는 보존하는 것이 정직한 설계.

#### 5.1.5. `StudyInternalController.forceCloseStudy()` — HTTP 어댑터

```java
@PostMapping("/{path}/force-close")
public ResponseEntity<StudyAdminResponse> forceCloseStudy(
        @PathVariable String path,
        @RequestHeader("X-Internal-Service") String internalService,
        @RequestHeader(value = "X-Account-Id", required = false) Long requesterId) {

    StudyAdminResponse response = studyInternalService.forceClose(path, requesterId);
    return ResponseEntity.ok(response);
}
```

**왜 요청 본문이 비어있는가**

이 엔드포인트는 "스터디를 강제 종료한다" 라는 **단일 동작**만 수행한다. 본문에 담을 파라미터가 없어도 동작이 명확하다. RESTful 관점에서 "상태 전이" 를 의도한 POST 는 **본문이 필수가 아니다**.

**왜 PATCH 가 아니라 POST 인가**

회원 권한 변경 때 겪은 RestTemplate 의 PATCH 미지원 문제와 동일한 이유. 전 계층 POST 로 통일해 **의존성 최소화**를 우선했다.

**관리자용 목록 엔드포인트도 함께**

같은 컨트롤러 내에 관리자용 목록 조회 엔드포인트도 있다:

```java
@GetMapping
public ResponseEntity<Page<StudyAdminResponse>> listStudiesForAdmin(
        @RequestHeader("X-Internal-Service") String internalService,
        @RequestParam(required = false) String keyword,
        Pageable pageable) {

    Page<Study> page;
    if (keyword == null || keyword.isBlank()) {
        page = studyRepository.findAll(pageable);
    } else {
        page = studyRepository.findByTitleContainingIgnoreCaseOrPathContainingIgnoreCase(
                keyword, keyword, pageable);
    }

    return ResponseEntity.ok(page.map(StudyAdminResponse::from));
}
```

**경로 매칭 우선순위** — 기존 `/recent` 와 새 `/` 가 충돌하지 않는 이유

Spring 의 경로 매칭은 **더 구체적인 것을 우선** 한다:

- `GET /internal/studies/recent` → 정확 매칭 → `getRecentStudies()` 로 라우팅
- `GET /internal/studies` → 루트 매칭 → `listStudiesForAdmin()` 로 라우팅

따라서 두 엔드포인트가 같은 클래스 안에 공존해도 충돌하지 않는다.

### 5.3. admin-frontend — 프레젠테이션 계층

#### 5.3.1. `AdminStudyClient.forceCloseStudy()` — 본문 없는 POST

```java
public AdminStudyDto forceCloseStudy(String path, Long accountId, String nickname) {

    String url = ADMIN_SERVICE_URL + "/api/admin/studies/" + path + "/force-close";

    HttpHeaders headers = buildAdminHeaders(accountId, nickname);
    HttpEntity<Void> entity = new HttpEntity<>(headers);  // body = null

    ResponseEntity<AdminCommonApiResponse<AdminStudyDto>> response = restTemplate.exchange(
            url, HttpMethod.POST, entity,
            new ParameterizedTypeReference<>() {}
    );

    AdminCommonApiResponse<AdminStudyDto> result = response.getBody();
    if (result == null || !result.isSuccess()) {
        String msg = (result != null) ? result.getMessage() : "응답이 비어있습니다";
        throw new IllegalStateException("admin-service 강제 비공개 실패: " + msg);
    }
    return result.getData();
}
```

**`HttpEntity<Void>` — 본문 없는 요청 표현**

Java 의 제네릭 타입으로 "본문 타입이 없음" 을 표현하는 관용법. `HttpEntity<Void>` 는 헤더만 있고 본문은 null 인 HTTP 엔티티를 만든다.

**Content-Type 헤더가 없어도 되는 이유**

본문이 없으니 **협상할 표현 (representation) 이 없다**. HTTP 스펙상 Content-Type 은 본문이 있을 때만 의미가 있다. 있어도 무해하지만 없는 편이 깔끔하다.

**`buildAdminHeaders()` — 헤더 빌드를 헬퍼로 분리**

```java
private HttpHeaders buildAdminHeaders(Long accountId, String nickname) {
    HttpHeaders headers = new HttpHeaders();
    headers.set("X-Account-Id", String.valueOf(accountId));
    headers.set("X-Account-Nickname", nickname == null ? "" : nickname);
    headers.set("X-Account-Role", "ROLE_ADMIN");
    return headers;
}
```

`listStudies` 와 `forceCloseStudy` 둘 다 같은 세 헤더를 주입해야 하므로 중복을 피해 헬퍼로 분리. 회원 관리의 `AdminInternalClient` 에서는 메서드가 둘뿐이라 인라인으로 두었지만, 스터디 관리에서는 구조가 **대칭적으로 발전**하므로 헬퍼가 더 낫다.

```
[study-service] "이미 종료된 스터디는 다시 종료할 수 없습니다."
       ↓ (Study.forceClose 가 RuntimeException 으로 던짐)
[StudyInternalService] IllegalArgumentException 으로 래핑 (메시지 유지)
       ↓ (GlobalExceptionHandler → 400 Bad Request + JSON body)
[Feign Client] FeignException.BadRequest 로 변환
       ↓
[admin-service] Exception 으로 전파 (admin-service 는 메시지만 보존)
       ↓
[RestTemplate in admin-frontend] HttpClientErrorException 으로 변환
       ↓
[AdminStudyClient] IllegalStateException("admin-service 강제 비공개 실패: " + msg) 래핑
       ↓
[AdminStudyController] catch (Exception e) → e.getMessage() 추출
       ↓
브라우저 JSON: { success: false, message: "강제 비공개 처리에 실패했습니다: ..." }
```

각 계층이 **메시지를 보존하면서 자기 컨텍스트만 덧붙인다**. 디버깅할 때 맨 끝 메시지만 봐도 **원인에서 현재까지의 전체 스택**을 대략 알 수 있다.

#### 5.3.3. `studies.html` — 복잡한 상태 배지 + 일방향 전환 UX

**핵심 HTML — 상태 배지 조합**

```html
<td class="cell-status">
    <span th:if="${s.closed}" class="badge badge-closed">종료됨</span>
    <span th:unless="${s.closed}" th:if="${s.published}" class="badge badge-published">공개</span>
    <span th:unless="${s.closed or s.published}" class="badge badge-draft">비공개</span>
    <span th:if="${s.recruiting and not s.closed}" class="badge badge-recruiting">모집중</span>
</td>
```

**조건 분기 테이블**

| closed | published | recruiting | 표시되는 배지 |
|--------|-----------|------------|-------------|
| false | false | false | "비공개" |
| false | false | true | "비공개" + "모집중" (recruiting 은 보통 published 전에는 false 여야 하지만, 방어적 표시) |
| false | true | false | "공개" |
| false | true | true | "공개" + "모집중" |
| true | any | any | "종료됨" (다른 것보다 우선) |

**th:unless 의 활용**

`th:unless="${s.closed}"` = `th:if="${not s.closed}"` 와 같지만 가독성이 좋다. 긍정 조건과 부정 조건을 Thymeleaf 의 전용 속성으로 구분하면 **논리의 방향**이 명확해진다.

**제목 링크 — 새 탭으로 일반 사용자 사이트 열기**

```html
<td class="col-title">
    <a th:href="@{|http://localhost:8080/study/${s.path}|}"
       target="_blank"
       rel="noopener noreferrer"
       class="study-title-link"
       th:text="${s.title}">스터디 제목</a>
</td>
```

**왜 새 탭인가**: 관리자가 "이 스터디 뭐 하는 거지?" 를 빠르게 확인할 수 있게 돕는다. 관리 흐름을 끊지 않기 위해 새 탭.

**`rel="noopener noreferrer"` 의 보안 의미**:
- `noopener`: 새 탭의 `window.opener` 가 원래 탭에 접근하지 못하게 차단 (피싱 방지)
- `noreferrer`: 새 탭에 Referer 헤더를 보내지 않음 (추적 방지)

두 속성은 외부 링크를 새 탭으로 열 때의 **표준 보안 권장사항**이다.

**왜 하드코딩 `http://localhost:8080` 인가**: 학습 프로젝트 단계라 개발 환경 고정. 프로덕션에서는 `application.yml` 의 `user-site-url` 같은 속성으로 분리해야 한다. 현재는 TODO.

**관리 컬럼 — 종료된 스터디는 버튼 없음**

```html
<td class="cell-actions">
    <button th:unless="${s.closed}"
            type="button"
            class="btn btn-danger btn-sm btn-force-close">
        강제 비공개
    </button>
    <span th:if="${s.closed}"
          class="muted-text" style="font-size: 0.85em; opacity: 0.6;">
        이미 종료됨
    </span>
</td>
```

**`btn-danger` — 위험한 작업의 시각 표현**

회원 관리에서는 파란색 `btn-primary` 를 썼지만, 강제 비공개는 **되돌릴 수 없는 위험한 작업**이므로 붉은 계열로 표시. 사용자 (관리자) 의 주의를 환기시킨다.

CSS:

```css
.btn-danger {
    background: rgba(239, 68, 68, 0.12);
    color: #ef4444;
    border: 1px solid rgba(239, 68, 68, 0.4);
}
```

투명도 0.12 의 은은한 배경으로 **너무 자극적이지 않으면서도** 붉은 계열임을 명확히 드러냈다.

**JS — 일방향 전환**

```javascript
function applyForceClose(row) {
    // 1. 행 data 속성 업데이트
    row.dataset.studyClosed = 'true';

    // 2. 상태 셀 교체 — 기존 배지들을 모두 지우고 "종료됨" 만 남김
    var statusCell = row.querySelector('.cell-status');
    statusCell.innerHTML = '<span class="badge badge-closed">종료됨</span>';

    // 3. 관리 셀 교체 — 버튼 제거, "이미 종료됨" 텍스트로 대체
    var actionsCell = row.querySelector('.cell-actions');
    actionsCell.innerHTML =
        '<span class="muted-text" style="font-size: 0.85em; opacity: 0.6;">이미 종료됨</span>';

    // 4. 시각적 힌트 — 행 전체에 살짝 어두운 배경
    row.classList.add('row-closed');
}
```

**회원 권한 변경의 `applyRoleChange` 와의 결정적 차이**

- `applyRoleChange`: 토글이므로 **반대 방향 버튼**을 새로 그림
- `applyForceClose`: 일방향이므로 **버튼 자체를 제거**, "이미 종료됨" 텍스트로 대체

이 차이가 사용자 멘탈 모델에 그대로 반영된다. 종료된 행에는 다시 클릭할 것이 없다는 것을 **UI 로 명확하게** 보여준다.

**`row-closed` 클래스 — 처리된 행 흐리기**

```css
tr.row-closed {
    opacity: 0.55;
    transition: opacity 0.3s ease;
}
```

행 전체에 `opacity: 0.55` 를 주어 **"이 행은 이미 처리됨"** 이라는 시각적 힌트를 준다. `transition` 으로 부드럽게 흐려져 사용자가 **변화를 느낀다**.

너무 어둡지도 (0.2~0.3 은 너무 사라진 느낌) 너무 밝지도 (0.8 은 차이 없음) 않은 **0.55** 가 "흐리되 읽을 수는 있다" 는 균형점이다.

**강제 비공개 확인 다이얼로그 — 강한 경고**

```javascript
var confirmMsg =
    '「' + title + '」 스터디를 강제 비공개 처리하시겠습니까?\n\n' +
    '이 작업은 되돌릴 수 없으며, 스터디가 종료 상태가 됩니다.';
if (!confirm(confirmMsg)) {
    return;
}
```

- 제목을 `「 」` 로 감싸 **어떤 스터디인지 명확**
- 빈 줄 하나 (`\n\n`) 로 **경고 메시지 분리**
- "이 작업은 되돌릴 수 없으며" 로 **비가역성 명시**

회원 권한 변경은 "바꾸시겠습니까?" 한 줄이면 충분했지만, 이 기능은 **강한 경고**가 필요하다.

**URL 인코딩 처리**

```javascript
var url = '/studies/' + encodeURIComponent(path) + '/force-close';
```

`path` 에 **한글/특수문자**가 포함될 수 있다 (예: "스프링-스터디"). `encodeURIComponent` 로 안전하게 인코딩:

- `스프링-스터디` → `%EC%8A%A4%ED%94%84%EB%A7%81-%EC%8A%A4%ED%84%B0%EB%94%94`

Spring 의 `@PathVariable` 은 자동으로 URL 디코딩하므로 서버에서는 원래 한글로 받는다.

## 6. State Machine 패턴 심층

### 6.1. Study 엔티티의 State Machine

Study 의 상태는 네 개 플래그 (`published`, `recruiting`, `closed`, `useBanner`) 와 시각 필드들로 표현된다. 그러나 실제 **유의미한 상태 전이**는 다음과 같다:

```
                    [생성]
                      │ published=false, closed=false, recruiting=false
                      ▼
                    [초안]
                      │ publish()  ───→ 이미 공개됐거나 종료되면 예외
                      ▼
                    [공개]   ←──→  startRecruit() / stopRecruit()  (1시간 쿨다운)
                      │              (published 여야 함)
                      │
                    close()
                      │ published 여야 함. closed 면 예외
                      ▼
                    [종료]   (되돌릴 수 없음)
                    
                    
            [관리자 강제 종료 경로]
            
                    [초안]
                      │
                 forceClose()
                      │  published 무관. closed 면만 예외
                      ▼
                    [종료]
                    
                    [공개]
                      │
                 forceClose()
                      │
                      ▼
                    [종료]
```

### 6.2. 각 상태 전이 메서드의 불변식

| 메서드 | 전제조건 | 동작 |
|--------|----------|------|
| `publish()` | `!closed && !published` | `published=true, publishedDateTime=now` |
| `close()` | `published && !closed` | `closed=true, closedDateTime=now` |
| `startRecruit()` | `canUpdateRecruiting()` (published + 1시간 쿨다운) | `recruiting=true, recruitingUpdatedDateTime=now` |
| `stopRecruit()` | `canUpdateRecruiting()` | `recruiting=false, recruitingUpdatedDateTime=now` |
| `forceClose()` | `!closed` (published 무관) | `closed=true, closedDateTime=now` |

각 메서드가 **자기 전제조건을 스스로 검사**한다. 서비스 계층이 잊어도 도메인 객체가 지킨다.

### 6.3. Setter 직접 호출을 허용했다면 어떻게 됐을까

```java
// 끔찍한 예
study.setClosed(true);
// closedDateTime 이 null 로 남는다
// 종료 시각을 기록하고 싶으면 서비스가 추가로 setter 를 호출해야 함
study.setClosedDateTime(LocalDateTime.now());
// 불변식 검사 없음 — 이미 closed 인 스터디도 또 설정 가능
```

이 코드는 **동작하지만 위험하다**:

1. 서비스 A 는 `setClosedDateTime` 을 호출하고 서비스 B 는 잊는다. 일관성 깨짐.
2. 이미 종료된 스터디에 `setClosed(true)` 를 또 호출하면 기록이 덮어쓰임.
3. `closed=true` 인 스터디에 `setPublished(false)` 를 호출해서 "공개 안 됐던 것처럼" 조작 가능.

**도메인 메서드 `forceClose()` 는 이 세 가지 위험을 모두 원천 차단**한다.

### 6.4. State Machine 의 이점

| 이점 | 설명 |
|------|------|
| **의도 명시** | `study.forceClose()` 는 "강제 종료한다" 는 의도가 드러남 |
| **원자성** | 상태 전이에 필요한 모든 필드 변경을 한 메서드 안에서 보장 |
| **불변식 보호** | 전제조건을 항상 검사 |
| **확장성** | 이벤트 발행, 감사 로그 등을 한 곳에 추가 가능 |
| **테스트 용이성** | State 전이별로 단위 테스트를 쓰기 쉬움 |

## 8. 학습 포인트 요약

### 8.1. 도메인 설계 (가장 중요)
- **State Machine 패턴** — Study 엔티티가 자기 상태 전이 규칙을 소유
- **불변식 검사** — 엔티티 메서드 안에서 전제조건 검사
- **일방향 전환 vs 토글** — 비즈니스 성격에 맞는 UX 결정
- **이력 보존** — `published` 를 되돌리지 않고 `closed` 만 설정 (Option A)
- **YAGNI** — 지금 필요 없는 "관리자가 닫았음" 플래그는 나중에

### 8.2. DTO 전략
- **관리자 전용 DTO** — 공개 DTO 와 분리하여 관리 필드만 담음
- **비정규화 활용** — `memberCount` 로 컬렉션 테이블 접근 회피
- **세 번 복사되는 미러 DTO** — 각 서비스가 자기 DTO 소유

### 8.3. Repository 설계
- **관리자 전용 검색 메서드** — 기존 "published=true, closed=false" 조건 없는 버전
- **`ContainingIgnoreCase`** — 대소문자 무시 LIKE 검색
- **null-safe 처리는 호출 측에서** — Repository 는 keyword 유무로 분기하지 않음

### 8.4. 서비스 계층
- **예외 래핑** — `RuntimeException` → `IllegalArgumentException` (500 → 400)
- **메시지 보존** — 엔티티의 사용자 친화적 메시지를 계층 따라 전달
- **`@Transactional` + Dirty Checking** — `save()` 없이 자동 UPDATE

### 8.5. HTTP 설계
- **요청 본문 없는 POST** — 단일 동작을 표현할 때 본문 생략
- **URL 경로에 의도 담기** — `/force-close` 가 "강제 비공개한다" 는 명확한 시맨틱
- **POST 통일** — PATCH 의존성 회피, 전 계층 일관성

### 8.6. UI/UX
- **상태 배지 조합** — Thymeleaf `th:if` / `th:unless` 로 State Machine 반영
- **일방향 전환의 시각화** — 버튼 제거 + opacity 0.55 + "이미 종료됨" 텍스트
- **강한 경고 다이얼로그** — "되돌릴 수 없음" 명시
- **URL 인코딩** — `encodeURIComponent` 로 한글 path 안전 처리
- **`rel="noopener noreferrer"`** — 새 탭 링크의 표준 보안 속성

### 8.7. 회원 권한 변경과의 비교로 배우는 것
- **설계 결정은 비즈니스 성격에 따라 달라진다** — 같은 5단계 패턴도 세부 규칙이 다름
- **"되돌릴 수 있느냐"** 가 UX 를 크게 바꾼다
- **자기 자신 검증의 유무** 는 "플랫폼 관리자의 권한 범위" 와 직접 연관

## 9. 미래 확장 포인트

### 9.1. Kafka 이벤트 발행 추가
현재는 Kafka 이벤트를 발행하지 않는다. 나중에 "관리자가 당신의 스터디를 차단했습니다" 알림을 보내려면:

```java
// StudyInternalService.forceClose() 에 추가할 코드
@Transactional
public StudyAdminResponse forceClose(String path, Long requesterId) {
    // ... 기존 검증 및 상태 전이
    
    try {
        study.forceClose();
    } catch (RuntimeException e) {
        throw new IllegalArgumentException(e.getMessage());
    }
    
    // [추가] Kafka 발행
    studyKafkaProducer.sendEvent(StudyEventDto.builder()
            .studyPath(study.getPath())
            .studyTitle(study.getTitle())
            .eventType("STUDY_FORCE_CLOSED")
            .accountId(null)  // 시스템 이벤트
            .build());
    
    return StudyAdminResponse.from(study);
}
```

notification-service 의 `StudyEventConsumer` 에 `STUDY_FORCE_CLOSED` 처리 분기를 추가하면 스터디 소유자와 멤버 전체에게 알림이 간다.

### 9.2. "관리자가 닫았음" 배지 (Option C 로 확장)
나중에 "일반 종료" 와 "관리자 강제 종료" 를 구분해서 표시하고 싶어지면:

```java
// Study 엔티티에 필드 추가
private boolean forcedClosed;

// forceClose() 에 한 줄 추가
public void forceClose() {
    if (this.closed) { throw ... }
    this.closed = true;
    this.closedDateTime = LocalDateTime.now();
    this.forcedClosed = true;   // ← 추가
}
```

DB 마이그레이션 (`ALTER TABLE study ADD COLUMN forced_closed BOOLEAN DEFAULT false`) 이 필요하고, `StudyAdminResponse` 와 관련 DTO 에도 필드를 전파해야 한다.

### 9.3. 감사 로그
`X-Account-Id` 를 이미 Feign 호출에 실어 보내므로, study-service 에 감사 로그 테이블만 추가하면:

```java
// 새 엔티티
@Entity
public class AuditLog {
    @Id @GeneratedValue
    private Long id;
    private String action;          // "FORCE_CLOSE_STUDY"
    private Long targetId;          // studyId
    private Long actorId;           // requesterId
    private LocalDateTime performedAt;
    private String detail;          // JSON 으로 추가 정보
}
```

`StudyInternalService.forceClose()` 에서 감사 로그 저장 한 줄만 추가하면 된다.

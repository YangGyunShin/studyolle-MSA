## 1. 기능 개요

### 1.1. 무엇을 하는 기능인가

플랫폼 관리자(ROLE_ADMIN) 가 회원 관리 페이지(/members) 에서 특정 회원의 권한을 `ROLE_USER` ↔ `ROLE_ADMIN` 으로 토글 변경할 수 있다. 페이지 새로고침 없이 fetch 로 즉석에서 행을 갱신하고, 토스트로 결과를 알려준다.

### 1.2. 왜 이 기능이 학습 가치가 높은가

단순한 CRUD 가 아니라 MSA 가 가진 **세 가지 핵심 주제**가 이 한 기능에 모두 녹아 있다.

| 주제 | 이 기능에서 드러나는 양상 |
|------|---------------------------|
| **서비스 간 동기 호출** | admin-frontend → admin-service → account-service 3-hop 동기 호출 체인 |
| **방어 깊이 검증** | 게이트웨이 / admin-service / account-service 세 곳에서 같은 검증을 반복 |
| **도메인 메서드 vs 데이터 setter** | `account.setRole()` 이 아닌 `account.changeRole()` 로 의도를 드러냄 |

## 2. 전체 호출 흐름 — 다섯 단계 네트워크 호출

```
[1] 브라우저 fetch
       POST /members/123/role
       body: { "role": "ROLE_ADMIN" }
       credentials: same-origin  (JWT 쿠키 자동 포함)
              │
              ▼ (admin-gateway :9080 의 JwtAuthenticationFilter 가 JWT 를 검증하고
                 X-Account-Id/Nickname/Role 헤더를 주입)
              │
[2] admin-frontend POST /members/{id}/role
       └─ AdminMemberController.updateMemberRole()
           · 권한 가드 (X-Account-Role == ROLE_ADMIN 확인)
           · 자기 자신 변경 시도 차단 (1차 방어선)
              │
              ▼
[3] AdminInternalClient.updateMemberRole()
       RestTemplate + lb://ADMIN-SERVICE (Eureka 해석)
       헤더 수동 주입: X-Account-Id, X-Account-Nickname, X-Account-Role
       본문: Map.of("role", "ROLE_ADMIN")
              │
              ▼
[4] admin-service POST /api/admin/members/{id}/role
       └─ AdminMemberController.updateRole()
           · 자기 자신 변경 시도 차단 (2차 방어선)
           · AccountAdminClient (Feign) 호출
              │
              ▼
[5] account-service POST /internal/accounts/{id}/role
       └─ AccountInternalController.updateRole()
           → AccountInternalService.updateRole()
                · 요청자 정보 검증 (X-Account-Id null check)
                · 자기 자신 변경 시도 차단 (3차 방어선)
                · role 화이트리스트 검증 (ROLE_ADMIN / ROLE_USER 만 허용)
                · Account 조회 → Account.changeRole() → Dirty Checking 으로 자동 UPDATE
              │
              ▼
[역순으로] AccountSummaryResponse → StudyAdminDto → AdminMemberDto → 브라우저 JSON
       │
       ▼
[6] 브라우저 fetch .then()
       · applyRoleChange(row, newRole) — 배지/버튼 즉석 갱신
       · showToast('...가 관리자로 변경되었습니다', 'success')
```

### 다섯 단계 네트워크 호출의 의미

모노리틱이었다면 `account.changeRole(newRole)` 한 줄이면 끝날 일이다. MSA 에서는 같은 일이 다섯 개의 HTTP 요청으로 분산된다. 이것이 MSA 의 **본질적 오버헤드**이며, 대신 각 서비스의 독립 배포·확장·장애 격리를 얻는다.

## 3. 방어 깊이(Defense in Depth) 3중 검증 구조

"자기 자신 권한 변경 금지" 라는 **단 하나의 규칙**을 세 곳에서 반복 검증한다.

```
┌─────────────────────────────────────────────────────────────┐
│ 1차 방어선 — admin-frontend (프레젠테이션)                   │
│   · 가장 빠르게 차단 → 불필요한 네트워크 호출 절약           │
│   · 장점: UX 빠름 (즉시 400 응답)                            │
│   · 약점: 개발자 도구로 fetch 직접 호출하면 우회 가능        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ (우회됨)
┌─────────────────────────────────────────────────────────────┐
│ 2차 방어선 — admin-service (Orchestration)                   │
│   · admin-gateway 의 AdminRoleFilter 를 우회한 경우 방어     │
│   · 내부 네트워크에서의 비정상 접근도 차단                   │
│   · 약점: admin-service 를 경유하지 않으면 의미 없음         │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ (우회됨)
┌─────────────────────────────────────────────────────────────┐
│ 3차 방어선 — account-service (최종 비즈니스 로직)            │
│   · 최후의 보루. 여기서 막지 못하면 진짜로 권한이 바뀜       │
│   · 도메인 규칙의 진정한 소유자                              │
│   · 다른 어떤 서비스가 호출해도 여기서는 반드시 검증됨       │
└─────────────────────────────────────────────────────────────┘
```

### 왜 이렇게까지 하는가 — "최소 권한 원칙" 과 "신뢰 경계"

한 서비스가 다른 서비스를 **완전히 신뢰하면** 그 서비스가 뚫리는 순간 연쇄 실패가 일어난다. 각 서비스가 자기에게 들어오는 요청을 **독립적으로** 검증하면, 상위 계층이 뚫려도 하위 계층이 방어할 수 있다.

이 원칙의 보상은 크다 — 단 하나의 규칙 ("자기 자신 권한 변경 금지") 을 위해 세 번 같은 코드를 쓰는 것은 **중복이 아니라 보험**이다.

## 4. 계층별 상세 구현

### 4.1. account-service — 최종 비즈니스 로직 계층

#### 4.1.1. `Account.changeRole()` — 도메인 메서드

```java
// Account.java
public void changeRole(String newRole) {
    this.role = newRole;
}
```

**왜 `setRole` 직접 호출이 아니라 별도 메서드인가**

Lombok `@Setter` 가 자동 생성한 `setRole()` 도 동작은 같다. 그러나 `setRole` 은 **"필드 값을 바꾼다"** 라는 단순 데이터 조작의 뉘앙스를 풍긴다. `changeRole` 은 **"권한을 변경한다"** 라는 비즈니스 행위를 명시적으로 표현한다.

같은 Account 엔티티의 `completeSignUp()` 도 본질은 두 필드 변경이지만 도메인 메서드로 감싸 의도를 드러낸 것과 같은 패턴이다.

**확장 여지 — 왜 지금 귀찮게 이렇게 하는가**

나중에 권한 변경 시 다음 기능이 추가되면:
- 변경 이력 기록 (`roleChangedAt` 필드 갱신)
- 이벤트 발행 (`RoleChangedEvent` 를 Kafka 로)
- 알림 전송 (당사자에게 "관리자가 되셨습니다")

이 모든 것을 `changeRole()` 메서드 안에 한 줄씩 추가하면 된다. `setter` 직접 호출 방식이라면 모든 호출 지점을 일일이 찾아 수정해야 한다.

**왜 여기서 유효성 검증(화이트리스트)을 하지 않는가**

`ROLE_ADMIN`/`ROLE_USER` 화이트리스트 검증은 **application 계층(Service)** 에서 한다. 도메인 객체는 이미 검증된 값을 받아 상태를 바꿀 책임만 진다. "어디까지 도메인이 책임지고 어디까지 application 이 책임지는가" 의 경계를 흐리지 않기 위함이다.

---

#### 4.1.2. `RoleUpdateRequest` — 요청 DTO (record)

```java
// account-service/src/main/java/com/studyolle/account/dto/request/RoleUpdateRequest.java
public record RoleUpdateRequest(String role) {
}
```

**왜 record 인가**

- 요청 본문은 **불변**이어야 한다.
- `equals/hashCode/toString` 이 자동 생성된다.
- getter 가 `role()` 형태로 자동 제공된다 (getRole() 이 아님 — record 는 접근자 규칙이 다름).
- setter 가 없어 컨트롤러에서 받은 후 **누가 값을 바꿔치기할 위험도 없다**.

**왜 `@NotBlank` 같은 검증 어노테이션이 없는가**

`AccountInternalService.updateRole()` 안에서 더 풍부한 도메인 검증 (`ROLE_ADMIN`/`ROLE_USER` 두 값만 허용) 을 수행하기 때문이다. 검증 로직을 한 곳에 모으는 편이 추적과 유지보수에 유리하다.

---

#### 4.1.3. `AccountInternalService.updateRole()` — 실제 비즈니스 로직

```java
@Service
@RequiredArgsConstructor
public class AccountInternalService {

    private final AccountRepository accountRepository;

    @Transactional
    public AccountSummaryResponse updateRole(Long targetId, Long requesterId, String newRole) {

        // 가드 1 — 요청자 정보 없음
        if (requesterId == null) {
            throw new IllegalArgumentException("요청자 정보가 없습니다.");
        }

        // 가드 2 — 자기 자신 권한 변경 금지 (3차 방어선)
        if (requesterId.equals(targetId)) {
            throw new IllegalArgumentException("자기 자신의 권한은 변경할 수 없습니다.");
        }

        // 가드 3 — role 값 화이트리스트
        if (!"ROLE_ADMIN".equals(newRole) && !"ROLE_USER".equals(newRole)) {
            throw new IllegalArgumentException("유효하지 않은 권한입니다: " + newRole);
        }

        Account account = accountRepository.findById(targetId)
                .orElseThrow(() -> new IllegalArgumentException("회원을 찾을 수 없습니다: id=" + targetId));

        account.changeRole(newRole);  // 도메인 메서드 호출

        return AccountSummaryResponse.from(account);
    }
}
```

**`@Transactional` 의 동작 메커니즘**

1. 메서드 진입 시 트랜잭션 시작 → 영속성 컨텍스트 생성
2. `accountRepository.findById()` 로 조회한 `account` 는 **영속 상태 (Managed State)** 가 된다
3. `account.changeRole(newRole)` 호출 → 엔티티 필드 변경
4. 메서드 정상 종료 시 트랜잭션 커밋
5. Hibernate 가 조회 시점 스냅샷과 현재 상태를 비교 (**Dirty Checking**) 하여 변경된 필드를 감지
6. 자동으로 `UPDATE account SET role=? WHERE id=?` 쿼리 발행

**이 과정에서 `save()` 호출이 필요 없다**. 영속 엔티티의 변경은 트랜잭션 커밋 시점에 자동으로 DB 에 반영된다.

**예외 롤백 정책**

검증 단계에서 `IllegalArgumentException` 이 던져지면 Spring 기본 정책 (`RuntimeException` 계열은 자동 롤백) 에 따라 트랜잭션이 롤백된다. 검증 단계에서는 실질적으로 변경된 게 없으니 롤백할 것도 없지만, **의미상 안전**하다.

**왜 `AccountInternalController` 에서 이 로직을 직접 하지 않는가**

기존의 조회 API 들은 Repository 를 직접 호출해도 괜찮았다. 그러나 이 메서드는 세 가지 비즈니스 규칙을 한 메서드 안에서 검증·실행해야 한다:

1. 요청자 정보 존재 여부 (`X-Account-Id` 헤더 필수)
2. 자기 자신 권한 변경 금지
3. `ROLE_ADMIN` / `ROLE_USER` 화이트리스트 검증

이런 검증 로직을 controller 에 두면 컨트롤러가 비대해지고 **단위 테스트**도 까다로워진다 (HTTP 맥락 없이 비즈니스 로직만 테스트하고 싶을 때 controller 는 불편한 단위다).

---

#### 4.1.4. `AccountInternalController.updateRole()` — HTTP 어댑터

```java
@PostMapping("/internal/accounts/{id}/role")
public ResponseEntity<AccountSummaryResponse> updateRole(
        @PathVariable Long id,
        @RequestBody RoleUpdateRequest request,
        @RequestHeader("X-Internal-Service") String internalService,
        @RequestHeader(value = "X-Account-Id", required = false) Long requesterId) {

    AccountSummaryResponse response = accountInternalService.updateRole(
            id, requesterId, request.role());
    return ResponseEntity.ok(response);
}
```

**컨트롤러가 하는 일은 단 네 가지**

1. HTTP 경로/헤더/본문 파싱
2. Service 호출
3. 결과를 `ResponseEntity` 로 감싸 반환
4. 끝

비즈니스 로직 (검증, 조회, 변경, DTO 변환) 은 **전부 service 안**에서 일어난다. 이렇게 단순 위임만 하는 controller 가 좋은 분리이며, 나중에 같은 비즈니스 로직을 다른 호출 경로 (예: 배치 작업, CLI 명령어) 에서 재사용할 때 그대로 가져다 쓸 수 있다.

**왜 `X-Account-Id` 를 controller 에서 받는가**

"요청자가 누구인가" 는 service 계층의 비즈니스 검증에 필요한 값이다. 하지만 service 가 `HttpServletRequest` 를 직접 보면 **layering 이 깨진다** (service 는 HTTP 를 몰라야 한다). 그래서 controller 에서 헤더를 꺼내 **일반 파라미터**로 변환해 service 에 넘긴다.

이렇게 하면 controller 는 **HTTP 와 도메인 사이의 통역자 역할**만 맡게 된다.

### 4.2. admin-service — Orchestration 계층

#### 4.2.1. `RoleUpdateRequest` — 자기 복사본

```java
// admin-service/src/main/java/com/studyolle/admin/dto/request/RoleUpdateRequest.java
public record RoleUpdateRequest(String role) {
}
```

**이 record 는 account-service 의 것과 완전히 동일하다. 왜 복사하는가**

MSA 원칙상 두 서비스는 서로의 클래스를 `import` 하지 않는다. 공유 라이브러리를 만들면 **"분산 모놀리스"** 가 되어 두 서비스가 공유 라이브러리 버전에 묶이고, MSA 의 독립 배포 이점이 사라진다.

Jackson 이 JSON 으로 직렬화/역직렬화할 때는 **필드 이름이 일치**하기만 하면 된다. "계약은 JSON 형태이지 Java 클래스가 아니다" — 이 관점이 모놀리스와 가장 다른 부분이다.

---

#### 4.2.2. `AccountAdminClient` — Feign Client

```java
@FeignClient(name = "account-service")
public interface AccountAdminClient {

    @GetMapping("/internal/accounts")
    PageResponse<AccountAdminDto> listAccounts(
            @RequestParam(required = false) String keyword,
            @RequestParam int page,
            @RequestParam int size,
            @RequestHeader("X-Internal-Service") String serviceName);

    @PostMapping("/internal/accounts/{id}/role")
    AccountAdminDto updateRole(
            @PathVariable("id") Long id,
            @RequestBody RoleUpdateRequest request,
            @RequestHeader("X-Internal-Service") String serviceName,
            @RequestHeader("X-Account-Id") Long requesterId);
}
```

**Feign 의 "마법" — 이 파일은 인터페이스일 뿐, 구현체가 없다**

`@EnableFeignClients` 를 만난 Spring 이 런타임에 **프록시 클래스를 자동 생성**한다. 이 프록시는 메서드 시그니처의 어노테이션을 읽어:

1. HTTP 메서드 결정 (`@GetMapping`, `@PostMapping`)
2. URL 경로 구성 (`@PathVariable`, `@RequestParam` 치환)
3. 요청 본문 직렬화 (`@RequestBody`)
4. 헤더 주입 (`@RequestHeader`)
5. 응답 역직렬화 (반환 타입)

개발자는 **"이 URL 에 이런 파라미터를 보내면 이런 응답이 온다"** 는 계약만 선언하고, HTTP 클라이언트 코드는 한 줄도 쓰지 않는다.

**`name = "account-service"` 가 의미하는 것**

Eureka 에 등록된 서비스 이름이다. Feign 내부의 LoadBalancer 가 이 이름을 **실제 인스턴스의 `host:port` 로 변환**한다. 덕분에 코드에 `localhost:8081` 같은 하드코딩이 등장하지 않는다.

account-service 가 여러 인스턴스로 수평 확장되어도 이 코드는 **전혀 수정할 필요가 없다**. 이것이 서비스 디스커버리의 힘이다.

**왜 `X-Internal-Service` 를 메서드 파라미터로 받는가**

Feign 의 `RequestInterceptor` 를 써서 모든 요청에 자동 주입할 수도 있다. 그러나 이 프로젝트의 다른 Feign Client 들 (`EventFeignClient`, `StudyFeignClient` 등) 이 모두 파라미터 방식을 택했다. 일관성이 중요하고, 호출 지점에서 **"어느 서비스 이름으로 나가는지"** 코드만 봐도 드러난다는 장점도 있다.

#### 4.2.3. `AdminMemberController` — Controller (2차 방어선)

```java
@RestController
@RequestMapping("/api/admin/members")
@RequiredArgsConstructor
public class AdminMemberController {

    private final AccountAdminClient accountAdminClient;

    @PostMapping("/{id}/role")
    public ResponseEntity<CommonApiResponse<AccountAdminDto>> updateRole(
            @PathVariable Long id,
            @RequestBody RoleUpdateRequest request,
            @RequestHeader("X-Account-Id") Long requesterId) {

        // 2차 방어선 — 자기 자신 권한 변경 금지
        if (requesterId.equals(id)) {
            return ResponseEntity.badRequest()
                    .body(CommonApiResponse.fail("자기 자신의 권한은 변경할 수 없습니다."));
        }

        AccountAdminDto updated = accountAdminClient.updateRole(
                id, request, InternalHeaderConstants.SERVICE_NAME, requesterId);

        return ResponseEntity.ok(CommonApiResponse.ok(updated));
    }
}
```

**"단순 위임 (pass-through)" 의 의미**

이 컨트롤러는 Feign 호출 결과를 거의 그대로 반환한다. **"그럼 admin-service 가 있으나 없으나 마찬가지 아니냐"** 는 의문이 들 수 있는데, 현재는 맞다. 그러나:

1. **경로/권한 경계 구분** — `/api/admin/**` 은 admin 전용 경로임을 URL 만 봐도 알 수 있다.
2. **집계 미래 대비** — 나중에 여러 서비스의 데이터를 조합하는 기능 (예: 회원 + 스터디 + 알림 통계) 이 추가되면, 이 서비스가 있어야 프론트가 **한 번의 호출로 집계 데이터**를 받을 수 있다.
3. **2차 방어선** — 게이트웨이가 뚫렸을 때의 보험.

단순 위임은 **구조를 미리 만들어 두는 투자**이지 낭비가 아니다.

**중첩 제네릭의 이유**

```java
ResponseEntity<CommonApiResponse<PageResponse<AccountAdminDto>>>
```

네 겹의 포장이 그대로 드러난다:

| 레이어 | 책임 |
|--------|------|
| `ResponseEntity` | HTTP 상태 코드와 헤더를 담는 Spring 의 HTTP 응답 래퍼 |
| `CommonApiResponse` | "success/message/data" 비즈니스 계약 래퍼 |
| `PageResponse` | 페이지네이션 메타데이터 래퍼 |
| `AccountAdminDto` | 실제 도메인 데이터 |

각 계층의 책임이 다르므로 **합쳐질 수 없는 구조**다. 못생겨 보이지만 이것이 정직한 표현이다.

### 4.3. admin-frontend — 프레젠테이션 계층

#### 4.3.1. `AdminInternalClient.updateMemberRole()` — RestTemplate 직접 호출

```java
@Component
@RequiredArgsConstructor
public class AdminInternalClient {

    private final RestTemplate restTemplate;
    private static final String ADMIN_SERVICE_URL = "lb://ADMIN-SERVICE";

    public AdminMemberDto updateMemberRole(
            Long memberId, String newRole, Long accountId, String nickname) {

        String url = ADMIN_SERVICE_URL + "/api/admin/members/" + memberId + "/role";

        HttpHeaders headers = new HttpHeaders();
        headers.set("X-Account-Id", String.valueOf(accountId));
        headers.set("X-Account-Nickname", nickname == null ? "" : nickname);
        headers.set("X-Account-Role", "ROLE_ADMIN");
        headers.setContentType(MediaType.APPLICATION_JSON);

        Map<String, String> body = Map.of("role", newRole);
        HttpEntity<Map<String, String>> entity = new HttpEntity<>(body, headers);

        ResponseEntity<AdminCommonApiResponse<AdminMemberDto>> response = restTemplate.exchange(
                url, HttpMethod.POST, entity,
                new ParameterizedTypeReference<>() {}
        );

        AdminCommonApiResponse<AdminMemberDto> result = response.getBody();
        if (result == null || !result.isSuccess()) {
            String msg = (result != null) ? result.getMessage() : "응답이 비어있습니다";
            throw new IllegalStateException("admin-service 권한 변경 실패: " + msg);
        }
        return result.getData();
    }
}
```

**왜 Feign 이 아니라 RestTemplate 인가**

admin-frontend 는 다른 일반 frontend 서비스와 일관성을 위해 **모든 외부 호출에 RestTemplate 을 사용**하는 방침을 택했다. Feign 을 혼용하면 "이 클라이언트는 Feign, 저 클라이언트는 RestTemplate" 식의 혼란이 생긴다.

대신 admin-service 쪽에서는 `AccountAdminClient` 를 Feign 으로 쓴다. 서버 간 호출 (admin-service → account-service) 은 Feign, 프레젠테이션 → 백엔드 호출 (admin-frontend → admin-service) 은 RestTemplate 이라는 **역할 구분**이 생긴다.

**왜 `lb://ADMIN-SERVICE` 로 호출하는가**

Eureka 에 등록된 서비스 이름을 `lb://` 스킴으로 참조하면, `@LoadBalanced` 가 붙은 RestTemplate 이 이 이름을 실제 `host:port` 로 변환한다. Feign 과 같은 서비스 디스커버리 혜택을 RestTemplate 에서도 받는 방법이다.

**왜 헤더를 수동으로 주입하는가**

일반적으로 브라우저가 `/api/admin/**` 을 호출할 때는 admin-gateway 의 `JwtAuthenticationFilter` 가 `X-Account-*` 헤더를 자동으로 붙여준다. 그러나 이 클라이언트는 **admin-frontend 의 JVM 안에서 직접 호출**하므로 게이트웨이를 거치지 않는다. 결국 이 클라이언트가 **손으로 헤더를 만들어 넣는다**.

이것이 차이점의 핵심이다:

| 호출 경로 | JwtAuthenticationFilter 경유? | 헤더 주입 주체 |
|----------|-------------------------------|----------------|
| 브라우저 → 게이트웨이 → admin-service | ✅ | 게이트웨이 자동 |
| admin-frontend (JVM 내부) → admin-service | ❌ | 이 클라이언트가 수동 |

**왜 `ParameterizedTypeReference` 를 쓰는가**

Java 는 런타임에 제네릭 타입이 **소거 (type erasure)** 된다. `List<String>` 은 런타임에 `List` 일 뿐, 제네릭 파라미터 정보가 사라진다.

`new ParameterizedTypeReference<>() {}` 는 **익명 서브클래스**를 만드는 트릭이다. 서브클래스는 부모의 제네릭 정보를 reflection 으로 들여다볼 수 있어, Jackson 이 정확한 타입 (`AdminCommonApiResponse<AdminMemberDto>`) 으로 역직렬화할 수 있다.

**`CommonApiResponse` 래퍼를 여기서 벗긴다 — Anti-Corruption Layer 패턴**

```java
if (result == null || !result.isSuccess()) {
    throw new IllegalStateException(...);
}
return result.getData();  // 래퍼 벗기기
```

Controller 에 넘어가는 것은 **순수 `AdminMemberDto`** 이지 `AdminCommonApiResponse` 가 아니다. 네트워크 계약 (CommonApiResponse) 을 경계에서 차단하면 내부 코드 (Controller, Template) 가 **깔끔**해진다.

이것이 **Anti-Corruption Layer** 패턴이다. 외부 서비스의 표현을 내 서비스의 내부 표현으로 변환하는 적응 층.

#### 4.3.2. `AdminMemberController.updateMemberRole()` — 1차 방어선 + JSON 응답

```java
@PostMapping("/members/{id}/role")
@ResponseBody
public ResponseEntity<Map<String, Object>> updateMemberRole(
        @PathVariable Long id,
        @RequestBody Map<String, String> body,
        @RequestHeader(value = "X-Account-Id", required = false) Long accountId,
        @RequestHeader(value = "X-Account-Nickname", required = false) String nickname,
        @RequestHeader(value = "X-Account-Role", required = false) String role) {

    // 권한 가드
    if (accountId == null || !"ROLE_ADMIN".equals(role)) {
        return ResponseEntity.status(HttpStatus.FORBIDDEN)
                .body(Map.of("success", false, "message", "권한이 없습니다."));
    }

    // 1차 방어선 — 자기 자신 권한 변경 금지
    if (accountId.equals(id)) {
        return ResponseEntity.badRequest()
                .body(Map.of("success", false, "message", "자기 자신의 권한은 변경할 수 없습니다."));
    }

    String newRole = body.get("role");
    if (newRole == null || newRole.isBlank()) {
        return ResponseEntity.badRequest()
                .body(Map.of("success", false, "message", "role 값이 필요합니다."));
    }

    try {
        AdminMemberDto updated = adminInternalClient.updateMemberRole(
                id, newRole, accountId, nickname);
        return ResponseEntity.ok(Map.of(
                "success", true,
                "role", updated.getRole(),
                "nickname", updated.getNickname()
        ));
    } catch (Exception e) {
        log.error("권한 변경 실패: memberId={}, role={}", id, newRole, e);
        return ResponseEntity.internalServerError()
                .body(Map.of("success", false, "message", "권한 변경에 실패했습니다: " + e.getMessage()));
    }
}
```

**`@ResponseBody` 로 JSON 을 직접 내려주는 이유**

브라우저 fetch 가 호출한다. **페이지 리다이렉트 대신 JSON 으로 응답**해야 JS 가 그 결과를 받아 행을 즉석에서 갱신할 수 있다. 이것이 **"권한 변경 후 페이지 깜빡임 없음"** UX 의 핵심이다.

만약 이게 일반적인 `@GetMapping` 반환처럼 뷰 이름 ("members") 을 반환하면, 브라우저는 그 뷰를 새 페이지로 받아 전체 화면을 다시 그린다. 그런 동작을 원하지 않는다.

**POST 인데 PATCH 가 아닌 이유**

초안에서는 "부분 수정" 시맨틱을 정확히 표현하기 위해 PATCH 를 선택했다. 그러나 Spring `RestTemplate` 의 기본 HTTP 클라이언트 (`SimpleClientHttpRequestFactory`) 가 **PATCH 메서드를 지원하지 않아** `Invalid HTTP method: PATCH` 에러가 발생했다.

해결 방안은 두 가지였다:
1. Apache HttpClient 5 의존성 추가 → PATCH 지원 → 의존성 증가
2. **POST 로 통일** → 의존성 최소 → RESTful 시맨틱 약간 손실

학습 프로젝트 특성상 **의존성 최소화가 우선**이고, URL 경로 (`/members/{id}/role`) 에 의도가 충분히 드러나므로 2번을 선택했다. 전 계층 POST 로 통일.

**에러 처리의 세 갈래**

```java
try {
    ...
    return 200 OK + success: true;
} catch (IllegalStateException e) {   // admin-service 에서 던진 비즈니스 에러
    return 500 + 사용자 친화적 메시지;
} catch (Exception e) {                // 네트워크/기타 예상 못한 에러
    return 500 + 일반 메시지;
}
```

JS 는 `response.ok` 와 `data.success` 두 가지를 모두 확인해 성공/실패를 구분한다.

#### 4.3.3. `members.html` — Thymeleaf + 이벤트 위임 + 토스트

**핵심 HTML 구조**

```html
<tr th:each="m : ${members}"
    th:attr="data-member-id=${m.id}, data-member-role=${m.role}">
    ...
    <td class="cell-role">
        <span class="badge"

    ...
    <td class="cell-actions">
        <!-- 본인 계정은 버튼 자리에 안내 텍스트 -->
        <span th:if="${m.id == currentAccountId}" class="muted-text">
            본인 계정
        </span>
        <!-- 본인이 아닐 때만 토글 버튼 -->

                data-target-role="ROLE_ADMIN">
            관리자로 승격
        </button>

                class="btn btn-ghost btn-sm btn-role-change"
                data-target-role="ROLE_USER">
            사용자로 강등
        </button>
    </td>
</tr>
```

**왜 `data-*` 속성을 사용하는가**

`data-*` 는 HTML5 표준 속성으로, JS 에서 `element.dataset` 으로 깔끔하게 읽을 수 있다. 전통적으로 `id` 에 값을 박거나 `class` 에 의미를 넣던 방식보다 훨씬 명시적이다.

| 속성 | 용도 |
|------|------|
| `data-member-id="${m.id}"` | fetch URL 구성 |
| `data-member-role="${m.role}"` | 현재 상태 추적 |
| `data-target-role="ROLE_ADMIN"` | 버튼이 "어떤 role 로 바꿀지" 박아둠 |

**왜 본인 계정 행은 서버 사이드에서 미리 분기하는가**

`th:if="${m.id == currentAccountId}"` 로 서버에서 **미리 렌더링을 분기**해두면:

1. **JS 가 꺼진 환경** (스크린리더, JS 비활성 브라우저) 에서도 본인 계정에는 위험한 버튼이 애초에 노출되지 않는다.
2. JS 에서 다시 필터링하는 로직이 필요 없다.
3. 악의적으로 HTML 을 수정하기 훨씬 어렵다 (렌더링 자체를 조작해야 함).

**JS — 이벤트 위임 패턴**

```javascript
document.addEventListener('click', function (event) {
    var btn = event.target.closest('.btn-role-change');
    if (!btn) return;   // 다른 버튼 클릭은 무시

    var row = btn.closest('tr');
    var memberId = row.dataset.memberId;
    var nickname = row.querySelector('.col-nickname').textContent.trim();
    var targetRole = btn.dataset.targetRole;

    // 확인 다이얼로그
    var roleLabel = (targetRole === 'ROLE_ADMIN') ? '관리자' : '일반 사용자';
    if (!confirm(nickname + ' 님을 ' + roleLabel + ' 로 변경하시겠습니까?')) {
        return;
    }

    // 중복 클릭 방지
    btn.disabled = true;
    var originalText = btn.textContent;
    btn.textContent = '처리 중...';

    fetch('/members/' + memberId + '/role', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        credentials: 'same-origin',
        body: JSON.stringify({ role: targetRole })
    })
        .then(function (res) {
            return res.json().then(function (data) {
                return { ok: res.ok, status: res.status, data: data };
            });
        })
        .then(function (result) {
            if (result.ok && result.data.success) {
                applyRoleChange(row, result.data.role);
                showToast(nickname + ' 님이 ' + roleLabel + ' 로 변경되었습니다.', 'success');
            } else {
                var msg = (result.data && result.data.message) || '알 수 없는 오류가 발생했습니다.';
                showToast(msg, 'error');
                btn.disabled = false;
                btn.textContent = originalText;
            }
        })
        .catch(function (err) {
            showToast('네트워크 오류로 권한 변경에 실패했습니다.', 'error');
            btn.disabled = false;
            btn.textContent = originalText;
        });
});
```

**왜 이벤트 위임 패턴을 쓰는가**

버튼마다 `addEventListener` 를 다는 대신 `document` 에 한 번만 달고 `event.target` 으로 클릭된 요소를 판별한다.

장점:
1. **행이 동적으로 추가되어도** 새 핸들러를 달 필요 없음 (페이지네이션에서 데이터만 바꿔도 OK).
2. 메모리 사용량 절감 (리스너 하나 vs N개).
3. 리스너 제거도 한 번이면 됨.

**행 즉석 갱신 함수**

```javascript
function applyRoleChange(row, newRole) {
    row.dataset.memberRole = newRole;

    var roleCell = row.querySelector('.cell-role');
    var isAdmin = (newRole === 'ROLE_ADMIN');
    roleCell.innerHTML =
        '<span class="badge ' + (isAdmin ? 'badge-admin' : 'badge-user') + '">'
        + (isAdmin ? 'ADMIN' : 'USER')
        + '</span>';

    var actionsCell = row.querySelector('.cell-actions');
    var nextRole = isAdmin ? 'ROLE_USER' : 'ROLE_ADMIN';
    var nextLabel = isAdmin ? '사용자로 강등' : '관리자로 승격';
    var nextClass = isAdmin ? 'btn-ghost' : 'btn-primary';
    actionsCell.innerHTML =
        '<button type="button" class="btn ' + nextClass + ' btn-sm btn-role-change" '
        + 'data-target-role="' + nextRole + '">'
        + nextLabel
        + '</button>';
}
```

**페이지 새로고침 없이 즉석 갱신하는 이유**

사용자 입장에서 **클릭 → 즉시 변화**가 보이면 반응성이 좋다. 다음 페이지 이동이나 검색 시점에 어차피 서버에서 최신 값을 다시 받아오므로 **화면과 DB 가 영구적으로 어긋날 위험은 없다**.

만약 여기서 `window.location.reload()` 를 호출하면 전체 페이지가 다시 로드되어 ① 네트워크 비용 ② 스크롤 위치 손실 ③ 깜빡임 이 모두 발생한다. SPA 스타일의 부분 갱신이 우월한 이유다.

## 5. 핵심 설계 결정과 그 배경

### 5.1. 왜 admin-frontend 가 admin-service 를 `lb://` 로 직접 호출하는가 — admin-gateway 우회

```
일반 사용자 흐름:
브라우저 → admin-gateway :9080 → admin-frontend :9000

admin-frontend 내부 호출:
admin-frontend → (admin-gateway 건너뜀) → admin-service :8082
```

admin-frontend 는 **자기가 서버 사이드 애플리케이션**이므로 "브라우저처럼" 게이트웨이를 거칠 필요가 없다. Eureka 에서 admin-service 를 찾아 직접 호출하는 것이 효율적이다.

**트레이드오프**: 게이트웨이의 `JwtAuthenticationFilter` 를 거치지 않으므로, `X-Account-*` 헤더를 **수동으로 주입**해야 한다.

### 5.2. 왜 전 계층 POST 로 통일했는가 — HTTP 메서드 결정

| 옵션 | 시맨틱 | 기술적 비용 |
|------|--------|------|
| PATCH | 정확 (부분 수정) | Apache HttpClient 5 의존성 추가 필요 |
| **POST** | URL 경로에 의존 | **없음** |
| PUT | 리소스 전체 교체 의미 | 시맨틱 왜곡 |

POST + URL 경로 (`/role`) 조합이 이 프로젝트에는 최적이었다. `/internal/accounts/{id}/role` 이라는 URL 자체가 "권한 필드를 변경한다" 라는 의도를 충분히 드러낸다.

### 5.3. 왜 admin-service 가 자체 DB 가 없는가 — Orchestration 패턴

admin-service 는 **데이터를 직접 보유하지 않는다**. 회원 정보는 account-service 가, 스터디 정보는 study-service 가 소유한다. admin-service 는 여러 서비스의 데이터를 **조합해서 화면에 필요한 형태로 가공** 한다.

이것이 **Orchestration (또는 BFF — Backend For Frontend)** 패턴이다.

장점:
- 관리자 기능 추가 시 각 도메인 서비스에 영향 없음 (도메인 서비스는 자기 책임만 수행).
- 데이터 소유권이 명확 (회원은 account-service 가, 항상).

단점:
- 모든 관리자 요청이 두 번의 네트워크 홉 (admin-frontend → admin-service → 도메인 서비스) 을 거침.
- 도메인 서비스의 `/internal/**` API 가 정비되어 있어야 가능.

### 5.4. 왜 도메인 메서드 `changeRole()` 인가 — setter 대신 도메인 언어

```java
// 나쁜 예
account.setRole("ROLE_ADMIN");

// 좋은 예
account.changeRole("ROLE_ADMIN");
```

단순해 보이는 차이지만 의도가 크게 다르다:

- `setRole` → "필드 값을 바꾼다" (데이터 조작)
- `changeRole` → "권한을 변경한다" (비즈니스 행위)

**확장 지점**을 하나로 모으는 것이 핵심이다. 나중에 감사 로그, 이벤트 발행, 알림 전송이 추가되면 `changeRole()` 한 곳에 모두 집어넣을 수 있다.

## 7. 학습 포인트 요약

### 7.1. MSA 패턴
- **Orchestration / BFF** — admin-service 가 자체 DB 없이 여러 서비스를 조합
- **서비스 디스커버리** — Eureka + `lb://` + `@LoadBalanced`
- **Feign Client** — 인터페이스 선언만으로 HTTP 호출 자동화
- **Anti-Corruption Layer** — `CommonApiResponse` 래퍼를 경계에서 벗기기

### 7.2. 보안 설계
- **Defense in Depth** — 한 규칙을 여러 계층에서 반복 검증
- **신뢰 경계** — 각 서비스가 들어오는 요청을 독립적으로 검증
- **게이트웨이 필터 분리** — `JwtAuthenticationFilter` + `AdminRoleFilter` + `OptionalJwtFilter`

### 7.3. 도메인 설계
- **도메인 메서드 vs setter** — `changeRole()` 이 `setRole()` 보다 우월한 이유
- **도메인과 애플리케이션의 책임 분리** — 엔티티는 상태 변경만, 검증은 Service
- **영속 엔티티 + Dirty Checking** — `save()` 없이 자동 UPDATE

### 7.4. 프레젠테이션 UX
- **이벤트 위임** — 동적 행에도 핸들러 자동 적용
- **낙관적 UI 갱신** — 응답 받자마자 화면 반영 (새로고침 없음)
- **`data-*` 속성 활용** — HTML5 표준 방식의 상태 저장
- **서버 사이드 분기 렌더링** — JS 꺼진 환경도 고려

### 7.5. 실용적 판단
- **POST 로 통일** — PATCH 시맨틱 손실 < 의존성 최소화
- **DTO 중복 복사** — MSA 독립 배포 원칙 ≫ DRY 원칙
- **단순 위임 컨트롤러** — 현재는 낭비 같지만 집계 기능 확장 대비 투자
